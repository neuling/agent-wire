## Findings

### Critical

**C1. [spec:45] Port 4040 in architecture diagram**
The ASCII diagram in the spec shows `:4040` twice (daemon binding and dashboard URL). The correct port is 4747.

**C2. [spec:42-47] "Streamable HTTP direct" path is a lie**
The spec describes a second connection mode: "Direct streamable HTTP MCP — clients that speak HTTP MCP connect to `http://127.0.0.1:4040/mcp`." This is false. The daemon's `/mcp` endpoint is an internal JSON dispatch protocol, NOT real MCP-over-HTTP. A Cursor or Windsurf instance attempting to point at `:4747/mcp` would get garbage. The truth is: any MCP client that supports stdio subprocesses works via `npx -y agent-wire-bridge`. The "How it works" section in README already says this correctly but adds a confusing sentence at the end: "HTTP MCP clients (Cursor, Windsurf, Codex, Continue) connect to `/mcp` directly" — this must be removed and replaced with the honest pull+piggyback framing.

**C3. [spec:325-334] Transport section "Streamable HTTP (direct)" subsection**
This entire subsection is false for the same reason as C2. It says "Clients that speak HTTP MCP connect to `http://127.0.0.1:4040/mcp` directly." Must be removed and replaced with honest framing.

**C4. [spec:267, spec:390] Port 4040 in Dashboard and Setup sections**
- Dashboard section: `http://127.0.0.1:4040/` (line 267)
- Dashboard section backing endpoints: `POST /mcp — the streamable HTTP MCP endpoint` (line 311) — this should be removed, it's the same false direct-HTTP claim
- Setup section step 4: `http://127.0.0.1:4040/` (line 390)

**C5. [README:141] "HTTP MCP clients connect to `/mcp` directly"**
README "How it works" prose ends with: "HTTP MCP clients (Cursor, Windsurf, Codex, Continue) connect to `/mcp` directly and pull via `wire_read`." This is false — they must use the stdio bridge. Must be removed.

### Important

**I1. [README:39] `npx` install line needs "not yet on npm" caveat**
The README says `claude mcp add --scope user agent-wire -- npx -y agent-wire-bridge` as if the package is on npm, but it is not yet. Need one line above: "*agent-wire is not yet on npm; until it is, replace `npx -y agent-wire-bridge` with the absolute path to `dist/bridge/index.js` in your clone.*"

**I2. [README + spec] Missing `wire_deregister` from tool list**
README "Tools" table (lines 111-120) lists 8 tools but is missing `wire_deregister`. The spec's "MCP Tools" section also has no `wire_deregister` entry. The tool exists in TOOL_DEFS (`src/bridge/index.ts` line 91), has a description, and should appear in both docs. The description should note it is called automatically by the bridge on exit.

**I3. [README + spec] Missing permissions allowlist section**
Neither the README nor the spec mentions that Claude Code prompts for permission on every `wire_*` tool call, and that adding `mcp__agent-wire__*` to `permissions.allow` in `~/.claude/settings.json` eliminates the friction. README needs a short "Skip permission prompts" section.

**I4. [README:171] Incorrect test count**
README Development section says `pnpm test # 40 tests (vitest)`. Actual count is 45 tests. Must update to `45 tests`.

**I5. [spec] Missing reaper description in "Auto-deregister" paragraph**
The spec's `wire_register` section says "the daemon tracks liveness via internal heartbeats from the bridge (every 10s, timeout 30s). When the bridge's stdio transport closes, the bridge stops heartbeating and the agent drops off the wire." This describes only the passive/implicit path. The active reaper (runs every 15s, removes agents with `last_activity` older than 30s) is not described. The spec should explicitly state: "A background reaper runs every 15s and removes any agent whose last heartbeat was more than 30s ago."

**I6. [spec] Missing graceful-disconnect description**
The spec does not mention that the bridge calls `wire_deregister` on stdio close, SIGINT, and SIGTERM. The Transport section or `wire_deregister` tool description should note this.

**I7. [spec] Universal piggyback not fully described**
The Delivery section says piggyback happens on `wire_status` and `wire_read`. It does not say that EVERY successful tool response carries `pending` when caller identity is known. The code does this universally (all tools except `wire_read` itself, where the response IS the pending). The spec should say "every successful tool response" not just `wire_status`.

**I8. [README:54] Confusing fallback description for push**
README Launching section says: "Without this flag, agent-wire still works — you just will not get push delivery into the session. Items still arrive via the piggyback mechanism on the next tool call (via `wire_read`)." The parenthetical "(via `wire_read`)" is misleading — piggyback is distinct from `wire_read`; it works on any tool call. Remove the parenthetical or clarify.

**I9. [spec:174] `wire_board_get` / `wire_board_patch` listed in MCP Tools section**
The spec has a `wire_board_get` / `wire_board_patch (v1.1, not v1)` blurb at the end of the Tools section. This adds noise — v1.1 tools that don't exist should be in Future Ideas, not in the Tools section.

**I10. [README + spec CLAUDE.md snippets] Not byte-identical**
The CLAUDE.md snippet in the spec (lines 363-385) and in the README (lines 63-88) differ in minor wording:
- Spec: `"a one-line description"` in wire_register bullet; README: same ✓
- Spec: `"Call `wire_register` with a short role-based name (e.g. "frontend-agent"),\n  your description, and `working_dir`."` — README has `"a one-line description, and your `working_dir`."` — minor discrepancy
- Both are acceptable, but they should be identical so users copy the same text. The README version is slightly more explicit ("a one-line description") and should be used as canonical.

**I11. [spec + README] `wire-claude` wrapper not mentioned in install/launch section of README**
README "Launching Claude Code" section only shows the raw `claude --dangerously-load-development-channels server:agent-wire` command. It doesn't tell users they can just run `wire-claude` instead. Should add a note.
